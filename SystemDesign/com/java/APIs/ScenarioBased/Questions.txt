1.A customer reports duplicate orders. Where would you start investigating?
   Sample Interview Answer

   "I would first determine whether the duplicate order was caused by the client, the network, the application, or downstream services. I'd trace the complete request flow from the API gateway to the database."

   Then investigate in this order:

   1. Check Whether the Client Sent Multiple Requests

   Questions:

   Did the user double-click the "Place Order" button?
   Did the mobile app retry?
   Did the browser resend after a timeout?
   Did the load balancer retry?

   Look at:

   API Gateway logs
   Access logs
   Request timestamps
   Request IDs

   Example:

   POST /orders
   10:10:15.123

   POST /orders
   10:10:15.500

   If two requests arrived from the client, the issue may be outside your service.

   2. Check for an Idempotency Key

   For order/payment APIs, every request should have something like:

   Idempotency-Key:
   abc123xyz

   Flow:

   Receive Request
         │
         ▼
   Check Redis/DB
         │
   Already Exists?
      │        │
    Yes       No
    │          │
   Return     Process Order
   Previous   Save Key
   Response

   If no idempotency mechanism exists, duplicate requests can create duplicate orders.

   3. Check Application Logs

   Trace the request using:

   Correlation ID
   Request ID
   Trace ID (e.g., OpenTelemetry/Zipkin)

   Questions:

   Was the controller called twice?
   Was the service method executed twice?
   Were there two database inserts?

   Example log:

   RequestId = abc123

   Controller
   ↓

   OrderService

   ↓

   Repository.save()

   If this sequence appears twice for the same request ID, investigate why.

   4. Check the Database

   Run queries like:

   SELECT customer_id,
          product_id,
          created_at,
          COUNT(*)
   FROM orders
   GROUP BY customer_id, product_id, created_at
   HAVING COUNT(*) > 1;

   Look for:

   Same customer
   Same cart
   Same payment
   Same order amount
   Nearly identical timestamps
   5. Check Database Constraints

   Is there a unique constraint that should prevent duplicates?

   Example:

   UNIQUE(idempotency_key)

   or

   UNIQUE(payment_id)

   Without appropriate constraints, concurrent requests may both insert rows successfully.

   6. Check Concurrent Requests (Race Condition)

   Two requests might reach the service at nearly the same time:

   Request A
             \
              Check "Order Exists" → No
   Request B /
               Check "Order Exists" → No

   ↓

   Both insert

   ↓

   Duplicate Orders

   Possible fixes:

   Database unique constraints
   Optimistic locking
   Pessimistic locking
   Idempotency keys
   7. Check Retry Mechanisms

   Look for automatic retries in:

   HTTP clients
   Message queues
   Circuit breakers
   Retry libraries

   Example:

   POST Order

   Timeout

   Retry

   Original request actually succeeded

   ↓

   Duplicate order

   A timeout does not necessarily mean the original operation failed.

   8. Check Message Queue Behavior

   If using systems like Kafka or RabbitMQ:

   Receive Message

   ↓

   Process

   ↓

   Crash before ACK

   ↓

   Message redelivered

   ↓

   Duplicate Order

   Consumers should be idempotent because "at least once" delivery can produce duplicates.

   9. Check Payment Flow

   Sometimes:

   Payment succeeds

   ↓

   Response timeout

   ↓

   Client retries

   ↓

   New order created

   This is another case where idempotency is essential.

   10. Check Deployment or Infrastructure Issues

   Verify:

   Multiple service instances
   Sticky session changes (if relevant)
   Load balancer retries
   Reverse proxy retries
   API Gateway retry policies
   Root Causes You Might Find
   Root Cause	Solution
   User clicked twice	Disable button / UI debounce
   Network retry	Idempotency key
   Client retry	Idempotency key
   API Gateway retry	Safe retry configuration
   Race condition	Locking or unique constraints
   Duplicate Kafka message	Idempotent consumer
   Missing unique index	Add database constraint
   Payment timeout	Idempotent payment processing
   What Interviewers Want to Hear (Senior-Level)

   A concise senior-level response could be:

   "I would trace the request end-to-end. First, I'd verify whether duplicate requests
   originated from the client by checking API gateway logs and correlation IDs.
   Next, I'd inspect the service logs to see if the request was processed twice or retried.
    I'd verify whether idempotency keys are implemented and enforced, then examine the database
    for duplicate inserts and ensure appropriate unique constraints exist. If the service is asynchronous,
     I'd investigate message redelivery and consumer idempotency. Finally, I'd review retry policies,
      timeouts, and any race conditions caused by concurrent requests. My goal is to identify whether
      the duplication originated at the client, network, application, database, or messaging layer."

3.How would you identify an N+1 query problem?
First, what is an N+1 Query Problem?

Suppose you have two entities:

@Entity
public class Department {

    @Id
    private Long id;

    private String name;

    @OneToMany(mappedBy = "department")
    private List<Employee> employees;
}
@Entity
public class Employee {

    @Id
    private Long id;

    private String name;

    @ManyToOne
    private Department department;
}

Assume the database contains:

Department
-----------
IT
HR
Finance

Employee
---------
John -> IT
David -> IT
Alex -> HR
Sam -> Finance
Code
List<Department> departments = departmentRepository.findAll();

for (Department department : departments) {
    System.out.println(department.getEmployees().size());
}

Looks harmless.

What Happens Internally?
Query 1
SELECT * FROM department;

Returns

IT
HR
Finance

Now inside the loop

department.getEmployees()

Hibernate loads employees lazily.

For IT

SELECT * FROM employee
WHERE department_id = 1;

For HR

SELECT * FROM employee
WHERE department_id = 2;

For Finance

SELECT * FROM employee
WHERE department_id = 3;

Total Queries

1 query for departments

+

3 queries for employees

=

4 queries

Formula

1 + N

If there are

100 departments

Then

1 + 100

=

101 SQL Queries

That's the N+1 problem.

How Do You Identify It?
1. Enable Hibernate SQL Logging (Most Common)

In application.properties

spring.jpa.show-sql=true
spring.jpa.properties.hibernate.format_sql=true

logging.level.org.hibernate.SQL=DEBUG
logging.level.org.hibernate.orm.jdbc.bind=TRACE

Run the API.

If you see

SELECT * FROM department;

SELECT * FROM employee WHERE department_id=1;

SELECT * FROM employee WHERE department_id=2;

SELECT * FROM employee WHERE department_id=3;

SELECT * FROM employee WHERE department_id=4;

That's an N+1 problem.

2. Look for Repeated SQL

Example logs:

SELECT employee ...

SELECT employee ...

SELECT employee ...

SELECT employee ...

SELECT employee ...

Same query repeated with different IDs.

Huge red flag.

3. Hibernate Statistics

Enable

spring.jpa.properties.hibernate.generate_statistics=true

Example output

Entity fetch count = 100

Query execution count = 101

Very useful in production debugging.

4. Use a SQL Profiler

Popular tools:

P6Spy
datasource-proxy
IntelliJ Database Console
Database query monitor
APM tools (New Relic, Dynatrace, AppDynamics)

These show exactly how many SQL statements your API executes.

5. Performance Symptoms

Suppose

Departments = 1000

Each employee query takes

5 ms

Total

1000 × 5

=

5000 ms

The API suddenly takes 5–6 seconds instead of a few hundred milliseconds.

How Do You Fix It?
Solution 1: JOIN FETCH (Most Common)

Repository

@Query("""
    SELECT d
    FROM Department d
    JOIN FETCH d.employees
    """)
List<Department> findAllWithEmployees();

Hibernate executes

SELECT *
FROM department d
JOIN employee e
ON d.id = e.department_id;

Only one query.

Solution 2: @EntityGraph
@EntityGraph(attributePaths = "employees")
List<Department> findAll();

Spring Data automatically fetches employees.

Solution 3: DTO Projection

Instead of loading entities

SELECT new DepartmentDTO(
    d.name,
    e.name
)

Only required columns are fetched.

Solution 4: Batch Fetching
spring.jpa.properties.hibernate.default_batch_fetch_size=50

Instead of

SELECT * FROM employee WHERE department_id = 1;
SELECT * FROM employee WHERE department_id = 2;
SELECT * FROM employee WHERE department_id = 3;

Hibernate may execute

SELECT *
FROM employee
WHERE department_id IN (1,2,3,...50);

Much fewer queries.

Interview Trick Question

Suppose someone changes

@OneToMany(fetch = FetchType.LAZY)

to

@OneToMany(fetch = FetchType.EAGER)

Will it solve the N+1 problem?

Answer: No, not necessarily.

EAGER means Hibernate tries to load the association automatically, but depending on the query,
it may still execute one query for the parent entities and additional queries for each child collection.
 EAGER does not guarantee a single SQL statement.

A JOIN FETCH or @EntityGraph explicitly tells Hibernate to retrieve the association in one query.

Interview Answer (1 Minute)

"I identify an N+1 query problem by enabling Hibernate SQL logs or using tools like P6Spy or Hibernate
Statistics. If I see one query fetching the parent entities followed by repeated queries fetching child
 entities with different IDs, that's a classic N+1 pattern. The repeated SQL and unusually high query
 count are strong indicators. I typically fix it using JOIN FETCH or @EntityGraph, and for some cases
 I use DTO projections or batch fetching depending on the use case."

 A concise decision table demonstrates mature thinking:

 Scenario	Recommended Approach
 5 employees per department	JOIN FETCH
 100 employees	JOIN FETCH or @EntityGraph (depending on the use case)
 10,000 employees	Separate paginated query, DTO projection, or batch fetching
 Dashboard showing only department names	Fetch only departments (no employee collection)
 Dashboard showing employee count	DTO projection with COUNT()
 REST endpoint returning department with all employees	Only if the collection is reasonably small;
  otherwise redesign the API to page the employees

  | Scenario                                              | Recommended Approach                                                                         |
  | ----------------------------------------------------- | -------------------------------------------------------------------------------------------- |
  | 5 employees per department                            | `JOIN FETCH`                                                                                 |
  | 100 employees                                         | `JOIN FETCH` or `@EntityGraph` (depending on the use case)                                   |
  | 10,000 employees                                      | Separate paginated query, DTO projection, or batch fetching                                  |

Why can "@Transactional" fail to roll back changes?

Interview Answer

"@Transactional is implemented using Spring AOP proxies. Rollback can fail if the method is not invoked through the proxy, if the exception type doesn't trigger rollback, or if the transaction boundary is configured incorrectly."

1. Exception is Caught (Most Common)
Problem
@Transactional
public void placeOrder() {
    try {
        paymentService.pay();
    } catch (Exception e) {
        System.out.println("Ignored");
    }

    orderRepository.save(order);
}

Since the exception is caught and not rethrown, Spring assumes the method completed successfully.

Result: Transaction commits.

Fix
@Transactional
public void placeOrder() {
    try {
        paymentService.pay();
    } catch (Exception e) {
        throw e;
    }
}

or

TransactionAspectSupport.currentTransactionStatus().setRollbackOnly();
2. Checked Exception

By default, Spring rolls back only for:

RuntimeException
Error

It does not roll back for checked exceptions.

Problem
@Transactional
public void placeOrder() throws IOException {
    throw new IOException();
}

Result: Transaction commits.

Fix
@Transactional(rollbackFor = IOException.class)

or

@Transactional(rollbackFor = Exception.class)
3. Self Invocation (Very Common)
Problem
@Service
public class OrderService {

    public void createOrder() {
        saveOrder();   // Internal call
    }

    @Transactional
    public void saveOrder() {
        // DB operation
    }
}

saveOrder() is called from the same class.

Spring proxy is bypassed.

Result: No transaction is created.

Fix

Call the method through another Spring bean.

@Service
public class OrderService {

    @Autowired
    private PaymentService paymentService;

    public void createOrder() {
        paymentService.saveOrder();
    }
}
4. Method is Private

Spring AOP cannot intercept private methods.

@Transactional
private void save() {}

Result: Transaction ignored.

5. Method is Final

Spring typically uses proxies to intercept methods.

@Transactional
public final void save() {}

Depending on the proxy mechanism, final methods may not be interceptable.

6. Object Created Using new
Problem
OrderService service = new OrderService();
service.save();

Not managed by Spring.

No proxy.

No transaction.

Correct:

@Autowired
private OrderService service;
7. Wrong Propagation

Example

@Transactional(propagation = Propagation.NOT_SUPPORTED)

This suspends any existing transaction.

Similarly, using REQUIRES_NEW creates a separate transaction, so its commit/rollback is independent of the caller.

Understanding propagation is important when explaining unexpected rollback behavior.

8. Asynchronous Execution
@Transactional
public void save() {

    CompletableFuture.runAsync(() -> {
        repository.save(...);
    });
}

or

@Async

The transaction is thread-bound.

New thread → New transaction context.

The async code won't participate in the original transaction unless explicitly designed to do so.

9. Multiple Data Sources

If you're using multiple databases:

DB1

DB2

and only one transaction manager is configured, operations on the other data source may not roll back with the first.

10. Transaction Already Committed

Once the transaction commits,

repository.save();

followed later by

throw new RuntimeException();

outside the transaction boundary cannot roll back changes that are already committed.

11. readOnly = true
@Transactional(readOnly = true)

Some JPA providers optimize for read operations. Although readOnly=true doesn't universally prevent writes,
 it's not intended for modifying data and can lead to unexpected behavior depending on the persistence
  provider.

12. Wrong Transaction Manager

Example:

@Transactional("mysqlTransactionManager")

while using a PostgreSQL repository.

The transaction manager must correspond to the resources being managed.

Summary Table
Scenario	Rollback?	Fix
RuntimeException	✅ Yes	Default behavior
Checked Exception	❌ No	rollbackFor
Exception caught and swallowed	❌ No	Rethrow or mark rollback-only
Self invocation	❌ No	Call through Spring bean
Private method	❌ No	Use public/protected method
new object	❌ No	Let Spring manage the bean
@Async / new thread	❌ No	Separate transaction or redesign
Wrong propagation	Depends	Choose correct propagation
Multiple transaction managers	Depends	Configure correctly
Transaction already committed	❌ No	Ensure exception occurs within transaction
Interview-Ready Answer (2 Minutes)

"@Transactional can fail to roll back for several reasons. The most common are: the exception is caught
and not rethrown, a checked exception is thrown without configuring rollbackFor, the transactional
method is called from another method in the same class (self-invocation), the method is private or
not invoked through a Spring-managed proxy, or the work happens in another thread using @Async or
CompletableFuture. Rollback behavior can also be affected by propagation settings, multiple transaction
managers, or exceptions occurring after the transaction has already committed. Since Spring implements
 @Transactional using AOP proxies, understanding the proxy and transaction boundaries is essential."

Interview-Ready Answer (2 Minutes)

"I would implement distributed tracing using a Trace ID or Correlation ID. When a request first
reaches the API Gateway, a Trace ID is generated or propagated from the client. That identifier
is forwarded to every downstream microservice using HTTP or messaging headers and included in all
application logs. I would use centralized logging to search by Trace ID and a distributed tracing
tool such as OpenTelemetry with Jaeger, Zipkin, or Grafana Tempo to visualize the request path and
latency across services. This allows me to identify where a request failed or which service introduced
the delay, even when the request spans synchronous REST calls and asynchronous messaging."

Follow-up Questions Interviewers Ask
1. What's the difference between a Correlation ID and a Trace ID?
Correlation ID: A business or application identifier used to correlate related logs. Its format is
application-defined.
Trace ID: A standardized identifier used in distributed tracing systems (for example, OpenTelemetry).
 A trace contains one or more spans, each representing an operation.
2. What is a Span?

A span represents a single unit of work.

Example:

Trace abc123

├── Gateway
├── Order Service
├── Payment Service
└── Database Query

Each operation is a separate span within the same trace.

3. How do you propagate the Trace ID automatically in Spring Boot?

With OpenTelemetry (or Spring Boot Micrometer Tracing in newer Spring Boot versions), instrumentation
automatically propagates trace context across supported HTTP clients, servers, and messaging frameworks.
 This reduces the need to manually add headers and keeps logs and traces correlated automatically.

If you had only 15 minutes to identify the root cause of a production incident, what would you check first?

Interview Answer

"With only 15 minutes, my goal is not to investigate every possibility but to quickly identify the component causing the issue and restore service if possible. I would first determine the scope of the incident, identify what changed, and use monitoring to locate the bottleneck before diving into detailed debugging."
